## 2. PgPool-II endpoint and backup role

    psql -h pgpool -p 9999 -U postgres -d vehicle_insurance -c 'SHOW POOL_NODES;'

Result:

    node_id | hostname      | status | pg_status | role    | pg_role | replication_state
    --------+---------------+--------+-----------+---------+---------+------------------
    0       | pg-primary    | up     | up        | primary | primary |
    1       | pg-standby-a  | up     | up        | standby | standby | streaming
    2       | pg-standby-dr | up     | up        | standby | standby | streaming

Command:

    SELECT rolname, rolsuper, rolcreatedb, rolcreaterole, rolreplication
    FROM pg_roles WHERE rolname='backup_operator';
    SELECT has_database_privilege('backup_operator','vehicle_insurance','CONNECT');
    SELECT has_schema_privilege('backup_operator','insurance','USAGE'),
           has_table_privilege('backup_operator','insurance.customers','SELECT');

Result:

    backup_operator|f|f|f|f
    t
    t|t

`backup_operator` is a login role for logical backup reads. It is not superuser,
cannot create databases or roles, and is not a replication role.

## 3. Full backup/restore scenario

Command on the test host:

    $env:PGPOOL_HOST_PORT='19999'
    $env:APP_HOST_PORT='18000'
    powershell -NoProfile -ExecutionPolicy Bypass \
      -File scripts/backup/Test-BackupRestore.ps1 \
      -EnvFile .env.example \
      -ProjectName vehicle-insurance-m5-test \
      -ComposeFiles docker-compose.yml,docker-compose.m5-test.yml \
      -BackupDirectory backups

Important result:

    [backup] removing stale M5 markers from previous interrupted test runs
    [backup] creating marker M5BR-20260626225811 through pgpool:9999
    [backup] pg_dump --format=custom through pgpool:9999 as backup_operator
    [backup] custom archive copied to ignored host path backups\vehicle_insurance-20260626-225811.dump (outside PGDATA volumes)
    [backup] deleting marker through pgpool:9999
    [backup] active database has no marker
    [backup] standby pg-standby-a also has no marker: DELETE replicated
    [backup] standby pg-standby-dr also has no marker: DELETE replicated
    [backup] creating separate vehicle_insurance_restore through pgpool:9999
    [backup] pg_restore only into vehicle_insurance_restore through pgpool:9999
    [backup] PASS: M5BR-20260626225811 is recovered only in vehicle_insurance_restore

What this proves:

- the control record existed before the dump;
- `pg_dump --format=custom` used `backup_operator` through `pgpool:9999`;
- the dump was copied to ignored host path `backups/`, outside the three PGDATA
  volumes `pg_primary_data`, `pg_standby_a_data` and `pg_standby_dr_data`;
- the record was deleted from the active database through PgPool;
- the logical delete replicated to both standby nodes;
- `pg_restore` targeted only `vehicle_insurance_restore`;
- `vehicle_insurance` was not restored over or overwritten.

The script removes its container copy and host dump in `finally`, so no dump
artifact remains after the evidence run.

## 5. Replication is not backup

The test intentionally deletes a dumped control record from the active database.
Both standby nodes show count `0`, which proves that physical streaming
replication copied the logical mistake/deletion. The independent custom dump
preserved the earlier logical state, and `pg_restore` recovered the record only
in the separate `vehicle_insurance_restore` database.

RPO for this mechanism is the time of the last successful dump. PITR/WAL archive
and pgBackRest remain optional project extensions and were not used for GATE-07.

- Deleted record recovered from backup: section 4.
