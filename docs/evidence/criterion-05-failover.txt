# Evidence: GATE-05 — Replication and Disaster Recovery (M3)
# Date: 2026-06-26
# Environment: isolated Docker Compose project `vehicle-insurance-m3-test`
# Isolation: temporary networks 172.40.0.0/24, 172.41.0.0/24 and 172.42.0.0/24;
#            the existing `vehicle-insurance-ha` containers, networks and volumes were not changed.

## 1. Fresh cluster: one primary and two standbys

Command (in `pg-primary`):

    repmgr -f /run/repmgr/repmgr.conf cluster show

Result:

    ID | Name          | Role    | Status    | Upstream
    ----+---------------+---------+-----------+------------
     1 | pg-primary    | primary | * running |
     2 | pg-standby-a  | standby |   running | pg-primary
     3 | pg-standby-dr | standby |   running | pg-primary

The command was executed as container user `postgres`; `repmgr` refuses root by design.

## 2. Physical streaming replication

Command (in `pg-primary`):

    SELECT application_name, client_addr, state
    FROM pg_stat_replication ORDER BY application_name;

Result:

    application_name | client_addr | state
    -----------------+-------------+-----------
    pg-standby-a     | 172.41.0.12 | streaming
    pg-standby-dr    | 172.42.0.11 | streaming
    (2 rows)

This confirms two live WAL receivers.

## 3. Data replication and read-only standby

Command on `pg-primary`:

    CREATE TABLE IF NOT EXISTS public.replication_probe(
      marker text PRIMARY KEY, created_at timestamptz DEFAULT clock_timestamp());
    INSERT INTO public.replication_probe(marker) VALUES ('M3-REAL-20260626');

Result:

    CREATE TABLE
    INSERT 0 1

Command on each standby:

    SELECT marker FROM public.replication_probe WHERE marker='M3-REAL-20260626';

Result on both `pg-standby-a` and `pg-standby-dr`:

    M3-REAL-20260626

Negative test on `pg-standby-a`:

    INSERT INTO public.replication_probe(marker) VALUES ('M3-NEGATIVE-WRITE');

Result:

    ERROR: cannot execute INSERT in a read-only transaction

## 4. Site-A fencing and manual DR promotion

Command on the test host:

    docker compose --env-file .env.example -p vehicle-insurance-m3-test \
      -f docker-compose.yml -f docker-compose.m3-test.yml stop pg-primary pg-standby-a
    docker compose --env-file .env.example -p vehicle-insurance-m3-test \
      -f docker-compose.yml -f docker-compose.m3-test.yml \
      ps --status running --services pg-primary pg-standby-a

Result of the second command: empty output. Both Site-A database containers were stopped
before promotion; this is the fencing confirmation.

Command (as `postgres` in `pg-standby-dr`):

    repmgr -f /run/repmgr/repmgr.conf standby promote

Result:

    waiting for server to promote.... done
    server promoted

Command:

    SELECT pg_is_in_recovery();

Result on `pg-standby-dr`:

    f

## 5. PgPool write after failover

After PgPool health checks detected the stopped Site-A backends, `SHOW POOL_NODES`
returned:

    node_id | hostname      | status | pg_status | role    | pg_role
    --------+---------------+--------+-----------+---------+----------
    0       | pg-primary    | down   | down      | standby | unknown
    1       | pg-standby-a  | down   | down      | standby | unknown
    2       | pg-standby-dr | up     | up        | primary | primary

Command through the unchanged application endpoint `pgpool:9999`:

    INSERT INTO insurance.customers(customer_number, first_name, last_name, national_id)
    VALUES ('DR-M3-REAL', 'Failover', 'Probe', '98765432109')
    RETURNING customer_number;

Result:

    DR-M3-REAL

## 6. Controlled rejoin of the former primary

The former `pg-primary` was kept fenced, its stopped container and its dedicated
test volume were removed, then it was re-cloned from `pg-standby-dr` with
`NODE_ROLE=standby`, `PRIMARY_HOST=pg-standby-dr`, and a new replication slot.

Relevant result from `repmgr cluster show` on the promoted DR:

    1 | pg-primary    | standby |   running | pg-standby-dr
    3 | pg-standby-dr | primary | * running |

The full command also reported `pg-standby-a` as unreachable, which is expected:
it intentionally remained stopped as the simulated lost Site-A node. The rejoined
former primary was healthy and its PostgreSQL log showed:

    started streaming WAL from primary ... on timeline 2
    NOTICE: standby node "pg-primary" (ID: 1) successfully registered

## GATE-05 coverage

- Physical streaming replication to a standby in DR: sections 1–3.
- Monitoring: `repmgr cluster show` and `pg_stat_replication`.
- Loss of the entire primary location: both Site-A database nodes stopped.
- Fencing before promotion: verified empty running-service list before `repmgr standby promote`.
- DR promotion and continued write path: sections 4–5.
- Safe, repeatable old-primary return: section 6 and `scripts/failover/Invoke-RejoinNode.ps1`.
