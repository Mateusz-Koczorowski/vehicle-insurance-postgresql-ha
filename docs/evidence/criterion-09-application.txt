# Date: 2026-06-26
# Environment: isolated Docker Compose project `vehicle-insurance-m6-test`

#            with non-default networks 172.45.0.0/24, 172.46.0.0/24,

# Secrets: passwords were loaded from `.env.example` as environment variables

## 1. Application scope

The placeholder application was replaced/extended as a small server-rendered
FastAPI/Jinja2 demo. The application has no login system, OAuth, cluster admin
panel or broad CRUD. It demonstrates only the database behavior required by M6:

- persona selection:
  - Anna Agent -> `app_agent_anna`;
  - Piotr Likwidator -> `app_adjuster_piotr`;
  - Ewa Audytor -> `app_auditor_ewa`;
- list views for customers, policies, claims and audit;
- allowed agent action: create a simple customer record;
- allowed adjuster action: update claim status, which records claim history;
- auditor read-only audit view;
- denied operations executed by PostgreSQL permissions, not by UI simulation:
  - agent attempts `INSERT` into `claims.payouts`;
  - adjuster attempts `UPDATE` on `insurance.policies`;
  - auditor attempts `UPDATE` on `insurance.customers`.

Every rendered screen shows the selected persona, `current_user`,
`inet_server_addr()` and `pg_is_in_recovery()`. The UI does not display
passwords or a full DSN.

## 2. PgPool-II endpoint

Application Compose environment:

```text
DATABASE_HOST=pgpool
DATABASE_PORT=9999
DATABASE_NAME=vehicle_insurance
```

The application code maps each persona to its login role and obtains only the
password from server-side environment variables. No path uses `postgres`, a
superuser or the database owner.
